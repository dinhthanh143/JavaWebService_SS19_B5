## PHẦN 1 - THIẾT KẾ KIẾN TRÚC HỆ THỐNG HR PORTAL

### 1. Sơ đồ luồng dữ liệu (Data Flow / Sequence Diagram)

Các sơ đồ dưới đây mô tả chính xác dòng chảy của dữ liệu, sự tương tác giữa Client trình duyệt, Bộ lọc an ninh Spring Security, Cơ sở dữ liệu và Tầng xử lý nghiệp vụ.

#### Kịch bản 1: Tiến trình Đăng nhập và Cấp cặp mã xác thực (Login)

Tiến trình này giải quyết bài toán trải nghiệm người dùng, cung cấp tấm vé ngắn hạn và dài hạn ngay khi nhân viên xác thực thành công.

```text
Client (Trình duyệt)            Spring Security / Auth Controller             Cơ sở dữ liệu (Database)
        |                                       |                                       |
        |--- 1. Gửi tài khoản, mật khẩu ------>|                                       |
        |                                       |--- 2. Kiểm tra tài khoản hợp lệ ----->|
        |                                       |<-- 3. Trả về thông tin nhân viên -----|
        |                                       |                                       |
        |                                       |--- 4. Tạo Access Token (15 phút) -----|
        |                                       |--- 5. Tạo Refresh Token (7 ngày) -----|
        |                                       |                                       |
        |                                       |--- 6. Lưu Refresh Token mới xuống ---->|
        |                                       |                                       |
        |<-- 7. Trả về cặp Token (JSON) --------|                                       |
        |                                       |                                       |
```

#### Kịch bản 2: Tiến trình dùng Refresh Token đổi Access Token mới (Token Refresh)

Tiến trình này diễn ra ngầm khi Access Token hết hạn, giúp nhân viên không bị văng khỏi hệ thống và không phải gõ lại mật khẩu.

```text
Client (Trình duyệt)             Bộ lọc an ninh / Auth Controller             Cơ sở dữ liệu (Database)
        |                                       |                                       |
        |--- 1. Gửi Refresh Token cũ ---------->|                                       |
        |                                       |--- 2. Tìm kiếm Refresh Token này ---->|
        |                                       |<-- 3. Trả về dữ liệu Token tìm được ---|
        |                                       |                                       |
        |                                       |--- 4. Kiểm tra thời hạn Token --------|
        |                                       |--- 5. Kiểm tra trạng thái Revoked ----|
        |                                       |                                       |
        |                                       |--- 6. Khởi tạo Access Token mới ------|
        |                                       |                                       |
        |<-- 7. Trả về Access Token mới --------|                                       |
        |                                       |                                       |
```

#### Kịch bản 3: Tiến trình thu hồi quyền truy cập lập tức (Revoke Token / Panic Button)

Tiến trình giải quyết nỗi đau bảo mật khi nhân viên nghỉ việc đột xuất. Admin hoặc chính nhân viên bấm nút để cắt đứt mọi phiên làm việc.

```text
Admin / Nhân viên                Auth Controller / Service                 Cơ sở dữ liệu (Database)
        |                                       |                                       |

        |                                       |                                       |

        |                                       |                                       |

        |                                       |       đen tạm thời nếu cần thiết      |
```

### 2. Chiến lược lưu trữ trạng thái Thu hồi (Revoke Strategy)

Để hệ thống hoạt động với hiệu năng cao nhất nhưng vẫn đảm bảo tính an ninh tuyệt đối, chiến lược lưu trữ trạng thái thu hồi được chia làm hai phần tương ứng với hai loại mã xác thực.

#### Đối với Refresh Token (Lưu trữ và kiểm tra trạng thái bền vững)

- **Vị trí lưu trữ:** Trạng thái thu hồi của Refresh Token được lưu trữ trực tiếp trong cơ sở dữ liệu quan hệ MySQL hoặc PostgreSQL tại bảng quản lý mã xác thực. Trường dữ liệu được thiết kế là một biến trạng thái kiểu đúng sai mang tên là `is_revoked`.
- **Cơ chế kiểm tra:** Khi Client gửi Refresh Token lên API làm mới để xin cấp Access Token mới, hệ thống sẽ thực hiện một câu lệnh truy vấn duy nhất theo mã Token đó. Nếu trường `is_revoked` mang giá trị là đúng, hệ thống lập tức từ chối và chặn đứng hành vi truy cập trái phép.

#### Đối với Access Token (Giải pháp kiểm tra tốc độ cao cho Token ngắn hạn)

- **Thách thức:** Access Token có đặc tính phi trạng thái và được kiểm tra ở mỗi request gửi lên các API chức năng như bảng lương hay hợp đồng. Nếu mỗi request đều phải truy vấn xuống cơ sở dữ liệu truyền thống để kiểm tra trạng thái thu hồi, hệ thống sẽ bị nghẽn cổ chai và giảm tốc độ xử lý nghiêm trọng.
- **Vị trí lưu trữ:** Hệ thống áp dụng một bộ lưu trữ phân tán có tốc độ đọc ghi trên bộ nhớ RAM là Redis làm nơi lưu trữ danh sách đen tạm thời.
- **Cơ chế kiểm tra nhanh nhất:** Khi nút bấm hoảng loạn được kích hoạt, mã Access Token hiện tại của nhân viên đó sẽ được đẩy vào Redis với cơ chế khóa tự hủy. Thời gian sống của bản ghi trên Redis được đặt chính xác bằng thời gian còn lại của Access Token đó.
- **Quy trình lọc:** Tại bộ lọc an ninh Spring Security Filter, hệ thống sẽ dùng lệnh kiểm tra sự tồn tại của khóa trên bộ nhớ RAM. Do Redis hoạt động trên RAM, tốc độ phản hồi chỉ mất dưới một mili giây, đảm bảo việc xác thực diễn ra mượt mà nhưng vẫn ngăn chặn được việc nhân viên bị sa thải cố tình lấy dữ liệu ngay trong phiên làm việc hiện tại.

### 3. Thiết kế giải pháp cho các Bẫy dữ liệu và Kịch bản ngoại lệ

Hệ thống quản lý nội bộ được cấu hình để không bao giờ bị sập hoặc trả về lỗi hệ thống không rõ ràng. Toàn bộ các ngoại lệ đều được bọc và trả về mã lỗi HTTP tiêu chuẩn đi kèm cấu trúc dữ liệu JSON rõ ràng bao gồm ba trường thông tin: thời gian xảy ra lỗi, mã trạng thái HTTP và thông báo chi tiết lý do từ chối.

#### Giải pháp cho Kịch bản 1: Client dùng Access Token đã hết hạn

- **Luồng xử lý:** Khi Access Token hết hạn, thư viện giải mã mã xác thực sẽ ném ra một ngoại lệ mang tên `ExpiredJwtException`. Bộ lọc an ninh sẽ bắt lấy ngoại lệ này ngay lập tức.
- **Kết quả trả về:** Hệ thống chặn không cho request đi vào tầng nghiệp vụ, trả về mã trạng thái HTTP là `401 Unauthorized` cùng thông báo lỗi rõ ràng hướng dẫn Client cần phải gọi API làm mới token để tiếp tục làm việc.

#### Giải pháp cho Kịch bản 2: Hacker dùng Refresh Token đã bị thu hồi trước đó

- **Luồng xử lý:** Khi nhận được yêu cầu cấp lại mã, hệ thống truy vấn vào cơ sở dữ liệu và tìm thấy bản ghi mã xác thực nhưng có trường `is_revoked` mang giá trị là đúng.
- **Kết quả trả về:** Hệ thống lập tức hủy bỏ tiến trình cấp mã, trả về mã trạng thái HTTP là `403 Forbidden` kèm theo thông báo cảnh báo mã xác thực này đã bị vô hiệu hóa bởi quản trị viên. Đồng thời hệ thống sẽ ghi lại log an ninh ghi nhận địa chỉ mạng thực hiện hành vi gian lận này.

#### Giải pháp cho Kịch bản 3: Client gửi Token sai định dạng hoặc bị sửa chữ ký

- **Luồng xử lý:** Thư viện giải mã sẽ thực hiện việc kiểm tra tính toàn vẹn của chữ ký số dựa trên chìa khóa bí mật của hệ thống. Nếu chữ ký bị lệch dù chỉ một ký tự hoặc chuỗi gửi lên không đúng cấu trúc ba phần của một Token, hệ thống sẽ ném ra ngoại lệ `MalformedJwtException` hoặc `SignatureException`.
- **Kết quả trả về:** Hệ thống kích hoạt bộ xử lý ngoại lệ an ninh toàn cục, trả về mã trạng thái HTTP là `400 Bad Request` kèm thông báo cấu trúc mã xác thực không hợp lệ, chặn đứng mọi nỗ lực tấn công dò mã hoặc giả mạo chữ ký của kẻ tấn công.
